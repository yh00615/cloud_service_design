# AWS Well-Architected Framework 체크리스트

이 문서는 QuickTable 3-Tier 아키텍처를 AWS Well-Architected Framework의 6가지 원칙에 따라 평가하기 위한 체크리스트입니다.

---

## 1. 운영 우수성 (Operational Excellence)

### 핵심 질문

#### 조직 문화
- [ ] 팀원들이 운영 우선순위를 이해하고 있습니까?
- [ ] 운영 절차가 문서화되어 있습니까?
- [ ] 정기적인 운영 검토 회의를 진행합니까?

#### 설계 원칙
- [ ] 코드로 운영을 수행합니까? (Infrastructure as Code)
- [ ] 작고 자주 변경합니까?
- [ ] 운영 절차를 정기적으로 개선합니까?

#### 모니터링
- [ ] CloudWatch를 사용하여 메트릭을 수집합니까?
- [ ] 알람을 설정하여 이상 징후를 감지합니까?
- [ ] 로그를 중앙 집중식으로 관리합니까?

### QuickTable 적용 사항

**현재 구성:**
- CloudWatch 메트릭 수집
- Auto Scaling 정책 설정
- ALB 액세스 로그 활성화

**개선 영역:**
- Infrastructure as Code (CloudFormation/Terraform) 도입
- 운영 Runbook 작성
- 정기적인 운영 검토 프로세스 수립

---

## 2. 보안 (Security)

### 핵심 질문

#### 자격 증명 및 액세스 관리
- [ ] IAM 역할을 사용하여 최소 권한 원칙을 적용합니까?
- [ ] MFA를 활성화했습니까?
- [ ] 루트 계정 사용을 제한합니까?

#### 탐지 제어
- [ ] CloudTrail을 활성화했습니까?
- [ ] GuardDuty를 사용하여 위협을 탐지합니까?
- [ ] Config를 사용하여 규정 준수를 모니터링합니까?

#### 인프라 보호
- [ ] 보안 그룹을 사용하여 트래픽을 제어합니까?
- [ ] 네트워크 ACL을 사용합니까?
- [ ] VPC Flow Logs를 활성화했습니까?

#### 데이터 보호
- [ ] 전송 중 데이터를 암호화합니까? (TLS/SSL)
- [ ] 저장 데이터를 암호화합니까? (RDS 암호화)
- [ ] 정기적으로 백업합니까?

### QuickTable 적용 사항

**현재 구성:**
- 계층화된 보안 그룹 (ALB-SG, Web-SG, DB-SG)
- HTTPS 종료 (ALB)
- RDS 암호화 활성화
- 프라이빗 서브넷에 애플리케이션 및 데이터베이스 배치

**개선 영역:**
- AWS WAF 적용 (SQL Injection, XSS 방어)
- Secrets Manager를 사용한 자격 증명 관리
- VPC Flow Logs 활성화
- GuardDuty 활성화

## 3. 안정성 (Reliability)

### 핵심 질문

#### 기초
- [ ] 서비스 한도를 모니터링합니까?
- [ ] 네트워크 토폴로지를 계획했습니까?

#### 워크로드 아키텍처
- [ ] 서비스 지향 아키텍처를 사용합니까?
- [ ] 느슨한 결합을 구현했습니까?

#### 변경 관리
- [ ] 시스템이 수요 변화에 적응합니까? (Auto Scaling)
- [ ] 리소스를 모니터링합니까?

#### 장애 관리
- [ ] 데이터를 백업합니까?
- [ ] 장애 격리를 구현했습니까?
- [ ] 재해 복구 계획이 있습니까?

### QuickTable 적용 사항

**현재 구성:**
- Multi-AZ 구성 (2개 가용 영역)
- RDS Multi-AZ (자동 장애 조치)
- Auto Scaling (수요 변화 대응)
- ALB 헬스 체크

**개선 영역:**
- 재해 복구 계획 수립 (RTO/RPO 정의)
- 정기적인 장애 복구 훈련
- 크로스 리전 백업

---

## 4. 성능 효율성 (Performance Efficiency)

### 핵심 질문

#### 선택
- [ ] 적절한 컴퓨팅 리소스를 선택했습니까?
- [ ] 적절한 스토리지 솔루션을 선택했습니까?
- [ ] 적절한 데이터베이스를 선택했습니까?

#### 검토
- [ ] 새로운 리소스 유형을 평가합니까?
- [ ] 성능을 정기적으로 검토합니까?

#### 모니터링
- [ ] 성능 메트릭을 수집합니까?
- [ ] 성능 저하를 감지합니까?

#### 트레이드오프
- [ ] 일관성, 내구성, 공간을 트레이드오프합니까?
- [ ] 캐싱을 사용합니까?

### QuickTable 적용 사항

**현재 구성:**
- ALB를 통한 트래픽 분산
- Auto Scaling을 통한 리소스 최적화
- RDS 읽기 전용 복제본 (선택사항)

**개선 영역:**
- ElastiCache Redis 도입 (세션 및 데이터 캐싱)
- CloudFront CDN 도입 (정적 콘텐츠)
- RDS Performance Insights 활성화
- 애플리케이션 성능 모니터링 (APM)

---

## 5. 비용 최적화 (Cost Optimization)

### 핵심 질문

#### 비용 인식
- [ ] 비용 할당 태그를 사용합니까?
- [ ] Cost Explorer를 사용합니까?
- [ ] 예산 알람을 설정했습니까?

#### 비용 효율적인 리소스
- [ ] 적절한 인스턴스 유형을 선택했습니까?
- [ ] Reserved Instances를 사용합니까?
- [ ] Spot Instances를 사용합니까?

#### 수요와 공급 일치
- [ ] Auto Scaling을 사용합니까?
- [ ] 사용하지 않는 리소스를 중지합니까?

#### 시간 경과에 따른 최적화
- [ ] 정기적으로 비용을 검토합니까?
- [ ] 새로운 서비스를 평가합니까?

### QuickTable 적용 사항

**현재 구성:**
- Auto Scaling (수요에 따른 리소스 조정)
- 리소스 태그 적용

**개선 영역:**
- Reserved Instances 구매 (예측 가능한 워크로드)
- Savings Plans 활용
- S3 Lifecycle 정책 (로그 아카이빙)
- Cost Anomaly Detection 활성화
- Trusted Advisor 권장 사항 검토

---

## 6. 지속 가능성 (Sustainability)

### 핵심 질문

#### 리전 선택
- [ ] 탄소 배출이 낮은 리전을 선택했습니까?
- [ ] 사용자와 가까운 리전을 선택했습니까?

#### 사용자 동작 패턴
- [ ] 사용 패턴을 분석했습니까?
- [ ] 불필요한 데이터 전송을 줄였습니까?

#### 소프트웨어 및 아키텍처 패턴
- [ ] 효율적인 코드를 작성합니까?
- [ ] 서버리스 아키텍처를 고려했습니까?

#### 데이터 패턴
- [ ] 데이터 분류 정책이 있습니까?
- [ ] 불필요한 데이터를 삭제합니까?

#### 하드웨어 패턴
- [ ] 최소한의 하드웨어를 사용합니까?
- [ ] Graviton 인스턴스를 고려했습니까?

### QuickTable 적용 사항

**현재 구성:**
- 서울 리전 사용 (사용자와 가까움)
- Auto Scaling (리소스 효율화)

**개선 영역:**
- Graviton 인스턴스 평가 (에너지 효율)
- Lambda를 사용한 서버리스 아키텍처 고려
- 데이터 보관 정책 수립
- 불필요한 로그 삭제

## 평가 방법

### 1. 각 원칙별 체크리스트 검토
- 체크리스트 항목을 하나씩 검토합니다.
- 현재 구성과 비교하여 체크합니다.

### 2. 위험 수준 평가
- **높음 (High)**: 즉시 조치 필요
- **중간 (Medium)**: 개선 계획 수립
- **낮음 (Low)**: 모니터링 및 검토

### 3. 개선 계획 수립
- 우선순위 결정
- 실행 계획 작성
- 담당자 지정

### 4. 정기적인 재평가
- 분기별 또는 반기별 재평가
- 아키텍처 변경 시 재평가
- 새로운 서비스 도입 시 재평가
